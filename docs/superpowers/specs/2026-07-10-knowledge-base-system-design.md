# 知识管理系统(资料问答)设计文档

- 日期:2026-07-10
- 状态:设计已评审通过,待写实现计划
- 范围:echo-agent-server(知识库服务)+ echo-agent-desktop(客户端入口)

## 1. 背景与目标

团队积累了大量异构资料(Word、Excel、音频、视频、纯文本),但没人完整读完过。
目标是让团队通过现有桌面客户端,基于这些资料提问并得到**正确、可核实**的回答。

四类问题都要支持:

- 精确事实查询(定位到原文条款/单元格/时间点)
- 理解与综合(跨文档总结、对比)
- 操作/流程指导(手册、SOP、培训视频里的"怎么做")
- 探索式提问("关于 X 我们有哪些材料")

纯 RAG 不足以覆盖这四类并保证正确率,因此采用**路线 C:Agentic 检索 + 强制校验**。

## 2. 非目标

- 不做云端方案:全链路纯内网私有化。
- Phase 1/2 不做全量 GraphRAG(压到 Phase 3 可选增强)。
- 不追求"拒答/转人工"级别的极高正确率,目标档位是"高:带引用可核实"。
- 本期不深度融入 echo-agent 的四层记忆/技能进化(做独立服务,留接口后接入)。

## 3. 关键约束与已确认决策

| 维度 | 结论 |
|---|---|
| 问题类型 | 四类全要,需可路由的混合检索,而非单一 RAG 管子 |
| 资料规模 | 万份以上且持续增长 → 中心化、增量索引、分层检索 |
| 正确率 | 高:每个关键结论带引用,客户端一键跳原文核实 |
| 部署 | 纯内网私有化,全本地模型,禁用云端 API |
| 数据构成 | Word/Excel/音视频/文本四类都重,需完整多模态摄取流水线 |
| Excel | 数字计算/汇总/条件筛选类问题多 → Text-to-SQL 为第一等公民 |
| 整合方式 | 独立知识库服务,接在 echo-agent-server,客户端直连,留口给 Agent |
| 技术路线 | 路线 C(Agentic 检索 + 校验),分阶段落地 |

**本地模型清单:** LLM(Qwen/DeepSeek,vLLM 部署)、Embedding(bge-m3)、
Reranker(bge-reranker)、ASR(FunASR)、OCR(PaddleOCR),均以 OpenAI 兼容接口暴露给 server。

## 4. 总体架构

三层结构,各司其职:

```
┌─────────────────────────────────────────────────────────┐
│  echo-agent-desktop (客户端)                              │
│  · 资料库页:上传/管理文件、看摄取进度、管权限分组          │
│  · 资料问答页:提问、看带引用的答案、点引用一键跳原文       │
└───────────────────────┬─────────────────────────────────┘
                        │ HTTP (JWT + 分组鉴权,复用现有)
┌───────────────────────▼─────────────────────────────────┐
│  echo-agent-server (知识库服务,新增模块)                 │
│  ┌─────────────┐  ┌──────────────────┐  ┌─────────────┐  │
│  │ 摄取流水线   │  │ 检索大脑(路线C)  │  │ 答案生成+校验│  │
│  │ (异步队列)   │  │ 路由→专用检索→补检│  │ 强制引用     │  │
│  └──────┬──────┘  └────────┬─────────┘  └─────────────┘  │
│  ┌──────▼─────────────────▼──────────────────────────┐  │
│  │ 存储:原文件 / 向量库(现有) / BM25 / 结构化表 / 元数据│  │
│  └────────────────────────────────────────────────────┘ │
└───────────────────────┬─────────────────────────────────┘
                        │ OpenAI 兼容接口(全本地)
┌───────────────────────▼─────────────────────────────────┐
│  本地模型运行时 (内网 GPU)                                │
│  LLM(Qwen/DeepSeek·vLLM) │ Embedding(bge-m3)             │
│  Reranker(bge-reranker)  │ ASR(FunASR) │ OCR(PaddleOCR)  │
└──────────────────────────────────────────────────────────┘
```

**职责边界:**

- 摄取流水线:把异构文件转成带坐标的知识单元并建索引(三条路线共用底座)。
- 检索大脑:路由 → 专用检索 → 迭代补检,是"路线 C 优于纯 RAG"的核心。
- 答案生成 + 校验:强制引用绑定 + 回原文校验 + 不足兜底。
- 存储层:向量库复用 server 现有能力;结构化表/元数据用关系库;原文件走文件系统/对象存储。
- 客户端只做入口:提问、渲染带引用答案、一键跳原文核实、上传与管理。

## 5. 摄取流水线

上传即入队,按格式分派解析器,产物统一成「带来源坐标的知识单元」:

```
上传文件 → 落对象存储 + 记元数据 → 算内容哈希(去重/变更检测) → 入队
   │
   ├─ Word/PDF ─→ 版面解析,保留页码/章节坐标;扫描件/图片 → PaddleOCR
   ├─ Excel   ─→ ① 每个 sheet 存成结构化表(schema:表名/列/类型/来源)
   │             ② 同时生成该表文本摘要("这张表讲什么")供语义发现
   ├─ 音视频   ─→ FunASR 转写,切成带 start/end 时间戳的片段
   └─ 文本/MD  ─→ 直接进
   │
   ↓
语义分块(保留坐标)→ 每块 = 一个知识单元
   ↓
索引:bge-m3 向量化 → 向量库;建 BM25;表 schema 注册到目录
   ↓
状态回写(客户端可看进度)
```

**两个关键设计点:**

1. **Excel 双表征** — 结构化表(给 Text-to-SQL 精确算数)+ 文本摘要(给语义检索"该用哪张表")。
   解决"数字算不准"和"先找到对的表"两个问题。
2. **坐标贯穿始终** — 每个知识单元携带精确来源坐标(文档页码/章节、Excel 的 sheet+单元格区间、
   视频时间戳区间)。这是"一键跳原文核实"的地基,摄取阶段不记就补不回来。

**增量与去重:** 内容哈希做变更检测,只重索引变更文件,`documents` 记版本号。

## 6. 数据模型(逻辑表)

| 表 | 关键字段 | 用途 |
|---|---|---|
| `documents` | id, 名称, 类型, 分组, 源路径, 哈希, 版本, 状态, 上传人 | 文件登记与增量 |
| `knowledge_units` | id, doc_id, 位置(json), 文本, 向量引用, 分组 | 检索基本单元(文档块/转写片段) |
| `tables` | id, doc_id, sheet, schema(json) | Excel 表目录,Text-to-SQL 读它选表 |
| `table_rows` | (按各表动态) | Excel 实际数据,SQL 在此执行 |
| `transcripts` | seg_id, doc_id, start_ms, end_ms, 文本 | 音视频转写(同时登记为 knowledge_unit) |
| `qa_logs` | 问题, 答案, 引用, 置信度, 用户反馈 | 评测 + 后续改进闭环 |

**分组隔离:** `group` 字段贯穿所有表,复用 echo-agent-server 现有组隔离,
摄取与检索两端都按 group 过滤。

## 7. 检索大脑(路线 C)

一次 `/kb/ask` 的流转:

1. **路由/拆解** — 轻量快模型判断问题类型、必要时拆子问题,决定调哪些检索器。
2. **专用检索器**(按需并行):
   - 文档类 → 向量 + BM25 混合召回 → bge-reranker 精排 → 带坐标片段
   - **表格类 → Text-to-SQL** → 在结构化表上写查询、执行、算出确切数字 + 来源单元格
   - 音视频类 → 转写文本检索 + 时间戳定位
   - 探索类 → 先列相关文档,再下钻
3. **迭代补检** — 编排器看召回质量,不足则换查询词再检(上限 2 轮,治 top-k 漏召回)。
4. **带引用生成** — 每个关键结论必须挂知识单元 ID。
5. **校验** — 逐条结论回原文比对;挂不上出处的标低置信度或删除。
6. **不足兜底** — 整体依据不足则返回"未找到充分依据"+ 最接近材料清单(服务探索式提问)。

**为何优于纯 RAG:** Text-to-SQL 把数字问题变成数据库确定性计算(消除数字幻觉);
迭代补检治 top-k 漏召回;路由让四类问题各走最优策略。复杂问题才启用多步,简单问题快速直答。

## 8. 正确率保障

对应目标档位"高:带引用可核实",五道防线层层收窄错误:

1. **重排降噪** — bge-reranker 把不相关片段挡在生成前。
2. **数字走确定性计算** — Excel 类用 Text-to-SQL,模型只写查询,算数交给数据库。
3. **强制引用绑定** — 生成时约束每个关键结论挂知识单元 ID,不允许无出处结论。
4. **回原文校验** — 生成后逐条比对引用原文,挂不上/对不上的标低置信度或剔除。
5. **不足兜底** — 依据不够不硬答,返回"未找到充分依据"+ 最接近材料。

**评测闭环:** 用 `qa_logs` + 人工整理的「黄金问答集」建评测集,量化三指标:
召回覆盖率、引用正确率、答案正确率(LLM 评判 + 人工抽检)。
改检索策略/换模型前后都跑,避免"改一处崩一片"。客户端答案带 👍/👎 + 纠正反馈,沉淀回 `qa_logs`。

## 9. 客户端集成

复用现有能力,不另起炉灶:

- **两个新页面**,走现有 `npm run new:page` 四件套约定(page + service + mock + 路由):
  - 资料库页:上传文件、看摄取进度、按分组管理
  - 资料问答页:提问 + 带内联引用的答案展示 + 反馈
- **复用现有基建**:axios 请求层 + urls 收口、service/mock 结构、JWT + 分组鉴权
  (直连 echo-agent-server)、i18next、electron-log。
- **一键跳原文核实**(引用可核实的落地):点答案里的引用 →
  - PDF/Word:打开到对应页码/章节
  - Excel:定位并高亮对应单元格区间
  - 音视频:跳到对应时间戳播放
  - 需要客户端一个轻量「来源查看器」组件,按类型分派。

## 10. 安全与权限

- 知识库服务是网络暴露接口,鉴权必须走 echo-agent-server 现有 JWT + 组隔离。
- 摄取与检索**两端都按 group 过滤**,团队成员只能问到自己分组权限内的资料,不能只在前端挡。
- 原始文件与转写文本可能含敏感信息,存储与日志按内网私有化要求处理,不外传。

## 11. 分阶段落地

**Phase 1 — MVP,核心问答跑通(覆盖四类问题基础版)**
- 摄取:Word/PDF/文本 + Excel 结构化 + 音视频转写,坐标贯穿
- 检索:Hybrid RAG + bge-reranker + Text-to-SQL 表格检索器
- 生成:强制引用 + 基础校验 + 不足兜底
- 客户端:资料库页 + 资料问答页 + 分组鉴权
- 已能答四类问题(探索式为基础版:靠混合检索 + 兜底列出相关材料清单),且数字类问题就准

**Phase 2 — 上"大脑"**
- 问题路由/拆解 + 迭代补检(治漏召回)
- 探索式"先列文档再下钻"检索器
- 完整校验 + 评测闭环 + 反馈沉淀

**Phase 3 — 可选增强**
- 局部 GraphRAG(仅当"全局摘要/关系类"成高频刚需时)
- 作为工具接入 echo-agent(留好的接口这时启用)

## 12. 测试策略

各层可独立验证:

| 层 | 怎么测 |
|---|---|
| 各格式解析器 | 给样例文件 → 断言产出知识单元和坐标正确(含扫描件 OCR、音视频时间戳) |
| Text-to-SQL | 给固定样例表 + 自然语言问题 → 断言 SQL 正确、算出数字正确(数字正确率护栏) |
| 检索质量 | 黄金问答集,量化召回覆盖率 / 引用正确率 / 答案正确率 |
| 端到端 | 完整 `/kb/ask` 流程 + 客户端引用跳转 |
| 权限隔离 | 跨分组提问必须检索不到越权资料 |

## 13. 依赖与前置

- 内网需备好 GPU 跑 vLLM(LLM)、bge-m3、bge-reranker、FunASR、PaddleOCR,
  并暴露 OpenAI 兼容接口给 server 配置(如 `ECHO_EMBED_URL` 等)。
- 这块是运维/环境依赖,不阻塞代码设计,但 Phase 1 启动前要确认算力到位。

## 14. 未决项与风险

- **算力评估**:万份+ 全本地摄取(尤其音视频转写、OCR)对 GPU 吞吐要求高,Phase 1 前需压测。
- **Text-to-SQL 鲁棒性**:真实 Excel 常有合并单元格、多表头、脏数据,解析成规整结构化表是难点,
  需要在 Phase 1 用真实样例验证解析器。
- **中文检索效果**:bge-m3 / bge-reranker 对中文资料的召回与精排效果需用黄金集实测调优。
- **增量与一致性**:文件更新后向量库、BM25、结构化表三处索引需同步更新,避免读到旧版本。
